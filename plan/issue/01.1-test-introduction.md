# #01.1 測試全面引入（backend 安全網，先於雲端遷移）

**Phase A — DB 層之後、Workers 化之前** ｜ 相依：#01（已實作 repo 層）｜ 後續：#02 起全程依賴

## 背景

目前**整個專案沒有任何自動化測試**（backend / frontend 皆無測試設定、無測試檔）。
而 #01 剛把 DB 層整個改寫成 Drizzle + repo 層，後續 #02（D1 driver）、#03（runtime 抽象）、
#04（R2）、#05（PR 輪詢）、#06（OG 上雲）每一步的核心驗收條件都是
**「self-host 行為零退化」**。沒有測試網，這個「零退化」只能靠人工點測，
每次雲端遷移都在裸奔。

本 issue 的定位是**在雲端遷移正式開始前，先把 backend 的安全網架好**：
建立 `bun test` 測試基礎建設、覆蓋純函式 / repo 層 / route 整合三層，
並接上 CI，讓之後 #02~#08 的每次改動都有迴歸保護。

## Decisions（主人已拍板，implementer 不需再問）

這些決定已定案，**不是開放問題，不要為此暫停等待人工輸入**：

1. **只做 backend**。frontend 測試留待 #08 或另開 issue。
2. **深度做到 route 整合**：純函式單元 + repo 層（隔離 SQLite）+ route（Hono `app.request()`）。
   外部依賴（GitHub / OpenAI / R2）一律 mock，不打真實網路。
3. **一併建立 CI**（GitHub Actions，push / PR 自動跑測試）。
4. **route 測試的 DB 隔離用 `DB_PATH` 暫存檔，本 issue 不做 db 注入式重構**；
   db 注入（`c.var.db`）與隨之而來的測試簡化**留給 #03**（在 #03 plan 補 follow-up）。
5. **coverage 只產報告、不當門檻**：CI 跑 `bun test --coverage` 並印出覆蓋率摘要供 review 參考，
   **不設 `coverageThreshold`、不因覆蓋率不足而擋 PR**。擋 PR 的只有「測試本身失敗」。
6. **測試目錄統一放 `backend/test/`**（含 `test/helpers/`），**不**使用 `src/**/__tests__`，避免兩處分裂。

## 需求摘要

- 為 backend 建立可重複執行、互不汙染、不依賴外部網路的 `bun test` 測試套件。
- 覆蓋三層：
  - **純函式單元**：`slugify`、`frontmatter` 等無 side effect 的 lib；`translator` 以 mock 處理（見下）。
  - **repo 層**：`repos/drafts`、`repos/presets`、`repos/images`，用隔離 SQLite 驗證 query 語意。
  - **route 整合**：透過 Hono `app.request()` 打 `/api/*`，驗證 status code / 回傳 JSON / DB 副作用。
- 外部依賴（GitHub REST、OpenAI、R2/S3）以 mock 隔離。
- 建立 GitHub Actions workflow，push 與 PR 時自動跑測試 + 印覆蓋率摘要（不擋）。
- 補上 `backend/package.json` 的 `test` / `test:coverage` script。

## 測試指令（明確定義，避免不一致）

| 用途 | 指令 | 行為 |
| --- | --- | --- |
| 本地快速跑（預設） | `bun test` | 跑全部測試，不算覆蓋率（最快） |
| 本地看覆蓋率 | `bun run test:coverage`（=`bun test --coverage`） | 跑測試 + 印覆蓋率摘要表 |
| CI | `bun run test:coverage` | 同上；**測試失敗才紅燈**，覆蓋率僅作報告不擋 |

- `package.json` scripts：`"test": "bun test"`、`"test:coverage": "bun test --coverage"`。
- CI 一律呼叫 `bun run test:coverage`（單一入口，本地與 CI 行為一致）。

## 假設

- 沿用 Bun 內建測試器（`bun test`，`bun:test` 的 `describe/test/expect/mock`），
  不額外引入 jest / vitest（符合 backend「用 Bun 原生 API」原則）。Bun 版本 1.3.10。
- repo 層測試用 `makeTestDb()`（in-memory SQLite + 共用建表函式），不碰 production DB。
- route 整合測試靠**啟動前設定 `DB_PATH` 指向暫存檔**讓 db 單例落在隔離 DB 上（見「route 測試隔離」）。
- 測試只跑 self-host（bun-sqlite）路徑；D1 路徑的測試屬 #02 範圍。

## 非目標 / 不在範圍

- frontend 測試（Vitest / RTL / 純函式）—— 留待 #08 或另開 issue。
- D1 driver 路徑的測試 —— 屬 #02。
- 把 route 的 db 單例重構成注入式 —— 屬 #03（本 issue 只用 `DB_PATH` 隔離，不動 production 結構）。
- E2E（真實啟動 server + 真實 GitHub/R2）—— 不做，外部依賴一律 mock。
- 效能 / 壓力測試。
- **coverage 門檻 / gate** —— 明確不做，只產報告（見 Decisions #5）。

## 允許的 production 微調（僅此清單，需在 PR 標明）

本 issue 原則上**不改 production 邏輯**，唯一允許的微調是為了「測試與 production 共用單一來源」：

- **`applySchema(sqlite)` 抽離（建議做）**：把 `db.ts` 目前寫死在 top-level 的建表 DDL +
  ALTER migration 抽成一個**純函式**（無 side effect），`db.ts` 與 `makeTestDb()` 都呼叫它。
  目的：避免測試另抄一份 DDL 與 `db.ts` 漂移。抽離後 `db.ts` 的 side effect 順序需不變
  （建目錄 → `applySchema` → export db）。需更新該函式 JSDoc 的 `@remarks` 記錄為何抽離。
- **translator helper 抽離（不建議，預設不做）**：`translator.ts` 目前只 export
  `isTranslationEnabled` 與 `translateDraft`，**沒有**純解析 helper。本 issue **不為了測試而拆它**；
  改用 mock fetch + env 的方式測（見下）。除非主人另外指示，否則不動 `translator.ts`。

## Issue 細節

### Goal

### route 測試隔離（關鍵地雷，務必照做）

`backend/index.ts` 在 module top-level **直接呼叫 `startPRChecker()`**，且其 default export 是
`{ port, fetch }`（不是 Hono 實例）。因此：

- **route 測試禁止 import `backend/index.ts`**（會啟動 PR 輪詢 setInterval、且拿不到可 `.request()` 的 app）。
- 改在 `test/helpers/testApp.ts` 自行組：`new Hono().route("/api", api)`，`api` 來自 `src/routes/api.ts`。
  這樣只掛 router、不觸發 `startPRChecker()`。
- **`DB_PATH` 必須在 import 任何會連帶載入 `src/lib/db.ts` 的模組之前設定**（`api.ts` → routes → `db.ts`，
  `db.ts` 於 import 時就 `mkdir` + 開檔 + 建表）。做法：在測試檔最頂端、所有受測 import 之前
  `process.env.DB_PATH = <tmp 檔>`；或用獨立的 setup 檔先設 env 再 `await import(...)` 動態載入。
- **避免跨檔 module-cache 汙染**：`db.ts` 的單例在整個 test process 內只建一次。若不同 route 測試檔需要
  各自乾淨的 DB，採其一：(a) 全部 route 測試共用同一個暫存 DB，並在 `beforeEach` 清空各表；
  或 (b) 用 `bun test` 的檔案隔離特性 + 每檔不同 `DB_PATH`（實作時驗證 Bun 是否每檔重新評估 module）。
  **預設採 (a)**（共用暫存 DB + `beforeEach` truncate），語意最可預期。
- 測試結束（`afterAll`）刪除暫存 DB 檔，確保不在 repo 留下 `data/`。

### route 覆蓋範圍（明確列舉，避免遺漏）

- **完整測**（純本地邏輯 + DB）：`drafts`、`slug`、`presets`、`images`（R2 部分 mock）。
- **mock 外部後測**：`translate`（mock OpenAI fetch）、`github`（mock `lib/github`）。
- **`upload` / `og`**：依賴 R2 與 OG 生成。本 issue 至少測「參數驗證 / 錯誤路徑」這層；
  若完整 happy-path 成本過高，可標 **deferred 到後續 issue** 並在本檔註明，不可無聲略過。

### Testing / verification strategy

- **純函式**：直接 import 呼叫，table-driven 測邊界（空字串、CJK、特殊字元、長度上限）。
- **translator**：`isTranslationEnabled` 切換 env 驗證；`translateDraft` 用 `mock.module()` 或覆寫
  `globalThis.fetch` 攔截 OpenAI 呼叫，env 於 import 前設定。
- **repo 層**：`makeTestDb()` 建乾淨 db → seed → 呼叫 repo 函數 → 斷言回傳與 DB 狀態；`beforeEach` 重建/清空。
- **route 整合**：`testApp().request("/api/...", { method, body })` → 斷言 `res.status`、`await res.json()`、
  以及 DB 副作用。外部呼叫用 `mock.module()` 攔截 `lib/github`、`lib/translator`、`lib/r2`。
- **CI**：`bun install --frozen-lockfile` + `bun run test:coverage`；測試失敗即擋，覆蓋率僅報告。

### Risks and mitigations

- **建表 DDL 重複（最大風險）**：DDL 寫死在 `db.ts` top-level。→ 抽 `applySchema(sqlite)` 純函式，
  `db.ts` 與 `makeTestDb` 共用單一來源（見「允許的 production 微調」）。
- **`db.ts` import 即有 side effect（`mkdir`/開檔/建表）**：route 測試 import app 會連帶執行。
  → `DB_PATH` 在所有受測 import **之前**設為 `os.tmpdir()` 下暫存檔，`afterAll` 刪除。
- **誤 import `index.ts` 啟動 `startPRChecker()`**：→ 一律用 `testApp` 從 `src/routes/api.ts` 組 app，
  禁止 import `index.ts`（已寫進「route 測試隔離」）。
- **跨檔共用 db 單例造成資料殘留**：→ 預設共用暫存 DB + `beforeEach` 清空各表。
- **bunfig coverage key 在 Bun 1.3.10 是否如預期**：→ implementer 須實測 `bun test --coverage` 確實印出
  摘要，且**未設門檻時不會因覆蓋率失敗**；若所選 key 行為與預期不符，以實測為準調整並在 PR 註明。
- **mock 漂移**：mock 介面與真實 `lib/github` 等簽章不一致會假綠。→ mock 緊貼簽章、用 `typeof` 對齊。
- **D1 無關**：本 issue 不測 D1；#02 換 driver 後另補（標記給 #02）。

### Notes for the implementer

- **先讀後改**：抽 `applySchema` 前 grep 所有 import `db.ts` 的地方，確認 side effect 順序不變。
- 測試一律放 `backend/test/`，不要用 `src/**/__tests__`。
- 外部依賴 mock 用 `bun:test` 的 `mock.module()`，放在測試檔頂端、import 受測模組之前。
- repo 測試的建表（`applySchema`）務必含 #01 migration 欄位（`github_path`/`github_sha`/`slug`），否則 slug 系列測試會炸。
- route 測試務必先設 `DB_PATH` 再 import；用 `testApp`，絕不 import `index.ts`。
- 依專案規範：為可測性微調函式（如抽 `applySchema`）要同步更新 JSDoc，`@remarks` 記錄 why。
- CI 用 `oven-sh/setup-bun@v2`、`working-directory: backend`、cache `~/.bun/install/cache`。
