# #01.1 測試全面引入（backend 安全網，先於雲端遷移）

**Phase A — DB 層之後、Workers 化之前** ｜ 相依：#01（已實作 repo 層）｜ 後續：#02 起全程依賴

## 背景

目前**整個專案沒有任何自動化測試**（backend / frontend 皆無測試設定、無測試檔）。
而 #01 剛把 DB 層整個改寫成 Drizzle + repo 層，後續 #02（D1 driver）、#03（runtime 抽象）、
#04（R2）、#05（PR 輪詢）、#06（OG 上雲）每一步的核心驗收條件都是
**「self-host 行為零退化」**。沒有測試網，這個「零退化」只能靠人工點測，
每次雲端遷移都在裸奔。

本 issue 的定位是**在雲端遷移正式開始前，先把 backend 的安全網架好**：
建立 `bun test` 測試基礎建設、覆蓋純函式 / repo 層 / route 整合三層，
並接上 CI，讓之後 #02~#08 的每次改動都有迴歸保護。

> 🎯 **主人已拍板的範圍決策（不需再問）**：
> 1. **本 issue 只做 backend**。frontend 測試留待 #08 或另開 issue。
> 2. **深度做到 route 整合**：純函式單元 + repo 層（in-memory/暫存 SQLite）+ route（Hono `app.request()`）。
>    外部依賴（GitHub / OpenAI / R2）一律隔離（mock 或邊界注入），不打真實網路。
> 3. **一併建立 CI**（GitHub Actions，push / PR 自動跑 `bun test`）。

## 需求摘要

- 為 backend 建立可重複執行、互不汙染、不依賴外部網路的 `bun test` 測試套件。
- 覆蓋三層：
  - **純函式單元**：`slugify`、`frontmatter`、`translator`（解析部分）等無 side effect 的 lib。
  - **repo 層**：`repos/drafts`、`repos/presets`、`repos/images`，用獨立的暫存 SQLite 驗證 query 語意。
  - **route 整合**：透過 Hono `app.request()` 打 `/api/*`，驗證 status code / 回傳 JSON / DB 副作用。
- 外部依賴（GitHub REST、OpenAI、R2/S3）以 mock 隔離。
- 建立 GitHub Actions workflow，push 與 PR 時自動 `bun install` + `bun test`。
- 補上 `backend/package.json` 的 `test` / `test:coverage` script。
- 產生並在 CI 報告測試**覆蓋率（coverage）**，設定一個起步門檻並可逐步往上 ratchet。

## 假設

- 沿用 Bun 內建測試器（`bun test`，`bun:test` 的 `describe/test/expect/mock`），
  不額外引入 jest / vitest（符合 backend「用 Bun 原生 API」原則）。
- repo 層測試用 `new Database(":memory:")` 或每測試一個暫存檔，套 `drizzle(...)` 後
  **手動套用與 `db.ts` 相同的建表 DDL**（見「注意 / 地雷」對 DDL 單一來源的處理）。
- route 整合測試在現況（route 直接 import db 單例）下，靠**啟動前設定 `DB_PATH` 指向暫存檔**
  讓單例落在隔離的 DB 上（見下方關鍵地雷）。
- 測試只跑 self-host（bun-sqlite）路徑；D1 路徑的測試屬 #02 範圍。

## 開放問題

  整合測試必須靠 `DB_PATH` 環境變數隔離。#03 會把 db 改成 `c.var.db`（middleware 注入），
  屆時整合測試可改用「建立測試 db → 注入 app」的乾淨寫法。
  → **建議**：本 issue 先用 `DB_PATH` 暫存檔方案讓測試跑起來；在 #03 的 plan 補一條
  「順手把 route 測試改為 db 注入」。**是否要在 #03 一併改、或本 issue 就先做一層薄抽象，
  請主人定調**（預設：本 issue 不改 production code 結構，只用 `DB_PATH` 隔離）。
- ~~覆蓋率門檻~~ → **已定調（主人拍板）**：本 issue **要 coverage**。用 `bun test --coverage`
  產生覆蓋率報告，CI 也跑帶 coverage。詳見下方「覆蓋率（coverage）策略」。
- ~~route 測試 db 注入~~ → **已定調（主人拍板）**：本 issue **不做** db 注入抽象，
  只用 `DB_PATH` 暫存檔隔離；db 注入式重構與測試簡化**留給 #03**（在 #03 plan 補 follow-up）。

## Issue 細節

### Title

#01.1 backend 測試全面引入 + CI（純函式 / repo / route 三層 + GitHub Actions）

### Goal

backend 有一套可重複、隔離、離線的 `bun test` 測試套件，覆蓋純函式、repo 層、route 整合，
並由 CI 自動執行；成為 #02 之後所有雲端遷移的迴歸安全網。

### Scope（影響 / 新增檔案）

- `backend/package.json` — 新增 `"test": "bun test"` 與 `"test:coverage": "bun test --coverage"` script。
- `backend/bunfig.toml` — 設定 coverage 開關與門檻（`[test] coverage` / `coverageThreshold`），
  讓本地與 CI 共用同一份門檻設定（單一來源，避免散落在 CI yaml）。
- 新增 `backend/src/lib/__tests__/` 或 `backend/test/`（擇一，全專案統一）：
  - `slugify.test.ts` — `slugify` / `isValidSlug` / `SLUG_PATTERN` 邊界（CJK→空字串、長度截斷、連字號收斂）。
  - `frontmatter.test.ts` — `parseFrontmatter` / `frontmatterToDraft` round-trip 與特殊字元。
  - `translator.test.ts` — 純解析 / 組裝部分（不打 OpenAI；網路呼叫部分 mock）。
- 新增 repo 層測試（每檔配一個建表 helper，套 in-memory / 暫存 SQLite）：
  - `repos/drafts.test.ts` — CRUD、`deleteDrafts` 批次、slug 衝突系列函數（`findSlugConflict` /
    `findSlugConflictExcludingIds` / `findSlugConflictBrief` / `findDraftsBySlug`）的 TRIM 語意、
    `listDrafts` 排序、`listSyncableDrafts` / `listPrOpenedDrafts` 過濾條件。
  - `repos/presets.test.ts` — presets CRUD。
  - `repos/images.test.ts` — list / upsert（同 key）/ insert。
- 新增 route 整合測試（Hono `app.request()`，外部依賴 mock）：
  - `routes/drafts.test.ts` — `GET/POST /drafts`、`GET/PATCH/DELETE /drafts/:id`、batch publish/delete。
  - `routes/slug.test.ts`、`routes/presets.test.ts`、`routes/images.test.ts`、`routes/translate.test.ts`。
  - GitHub / R2 / OpenAI 相關 endpoint 以 mock 隔離外部呼叫，只驗證 route 邏輯與 DB 副作用。
- 新增 `backend/test/helpers/`（測試輔助）：
  - `makeTestDb.ts` — 建立隔離的 Drizzle 實例 + 套用建表 DDL，回傳 `db`（給 repo 測試用）。
  - `mocks.ts` 或各自 `mock.module(...)` — GitHub / OpenAI / R2 的測試替身。
- 新增 `.github/workflows/test.yml` — push / PR 時 `oven-sh/setup-bun` → `bun install` →
  `bun test --coverage`（working-directory: backend）；coverage 未達門檻即紅燈。

### 覆蓋率（coverage）策略（主人已拍板：要 coverage）

- 用 Bun 內建覆蓋率：`bun test --coverage`（不額外引入 nyc / c8）。
- 門檻寫在 `backend/bunfig.toml` 的 `[test]` 區段（`coverageThreshold`），**本地與 CI 同一份來源**，
  低於門檻時 `bun test --coverage` 直接非零退出 → CI 紅燈擋 PR。
- **起步門檻採務實值、可逐步 ratchet**（避免一上來就卡死）：
  - 建議起步 line/function ≈ **70%**（純函式 + repo 層本來就容易拉高，route 整合再補）。
  - 每次 PR 只能往上調、不往下；達到穩定後再考慮拉到 80%+。
  - **起步數字本身可調**，請 implementer 以「第一版測試實際跑出來的覆蓋率」往下取一個略低的整數當門檻，
    確保引入當下是綠燈、之後只升不降。
- coverage 範圍**排除測試檔自身與純設定檔**（`bunfig.toml` 的 ignore 設定），聚焦受測 production 程式。
- CI log 需印出 coverage 摘要表，方便 review 時看趨勢。

### Out of scope

- 改動任何 production 程式邏輯（除非為了可測性的最小調整，且需在 PR 說明）。
- frontend 測試、D1 測試、coverage 門檻、E2E。

### Dependencies

- #01（Drizzle + repo 層）已完成 —— 本 issue 直接針對 repo 函數與 route 撰寫測試。

### Suggested implementation order

1. 加 `test` script + 建測試目錄慣例（決定 `__tests__` 或 `test/`，全專案統一）。
2. **純函式測試先行**（slugify / frontmatter / translator）—— 最快見綠燈，建立信心。
3. `makeTestDb` helper + repo 層測試 —— 確立「建表 DDL 在測試怎麼來」這個地基。
4. route 整合測試（含 `DB_PATH` 隔離與外部依賴 mock）。
5. GitHub Actions workflow，本地先用 `act` 或直接開 PR 驗證綠燈。
6. 補 JSDoc / README 測試章節（如何跑、如何加測試）。

### Acceptance criteria

      route 整合（drafts/slug/presets/images/translate）。
- [ ] 測試**不依賴外部網路**：GitHub / OpenAI / R2 全部 mock，斷網也能跑過。
- [ ] repo 測試用獨立 SQLite，**不碰** production `data/blog-editor.db`。
- [ ] route 整合測試以 `DB_PATH` 指向暫存檔隔離，測試結束清理。
- [ ] `.github/workflows/test.yml` 在 push / PR 觸發並跑過 `bun test`。
- [ ] `backend/package.json` 有 `test` 與 `test:coverage` script。
- [ ] `bun test --coverage` 能產生覆蓋率報告；`bunfig.toml` 設有 `coverageThreshold`，
      低於門檻時測試非零退出（CI 會紅燈）。
- [ ] CI workflow 跑的是 `bun test --coverage` 並印出覆蓋率摘要。
- [ ] 至少涵蓋 #01 改寫過的高風險點：slug 衝突 TRIM 語意、batch delete/publish、
      `listDrafts` 排序、`listSyncableDrafts` 過濾 —— 這些正是雲端遷移最怕退化的查詢。

### Testing / verification strategy

- **純函式**：直接 import 呼叫，table-driven 測邊界（空字串、CJK、特殊字元、長度上限）。
- **repo 層**：`makeTestDb()` 建乾淨 db → seed 資料 → 呼叫 repo 函數 → 斷言回傳與 DB 狀態。
  每個 `test`/`describe` 用新 db（in-memory）或 `beforeEach` 重建，確保隔離。
- **route 整合**：`app.request("/api/...", { method, body })` → 斷言 `res.status`、`await res.json()`、
  以及對 DB 的副作用（再查一次 repo / 直接查 db）。外部呼叫用 `mock.module()` 攔截
  `lib/github`、`lib/translator`、`lib/r2`。
- **CI**：workflow 跑 `bun install --frozen-lockfile` + `bun test`；失敗即紅燈擋 PR。

### Risks and mitigations

- **建表 DDL 重複（最大風險）**：DDL 目前寫死在 `db.ts` 的 top-level（且含 `await mkdir` 等
  side effect，import 即執行）。測試若另抄一份 DDL，會與 `db.ts` **漂移**（某天改欄位忘了同步）。
  → **緩解**：優先把建表 DDL 抽成一個**純函式** `applySchema(sqlite)`（無 side effect），
  `db.ts` 與 `makeTestDb` 都呼叫它（單一來源）。這是本 issue **唯一建議的 production 微調**，
  需在 PR 標明；若主人不想動 `db.ts`，退而求其次在 helper 內維護 DDL 並加註解警示同步。
- **`db.ts` 的 top-level `await mkdir` side effect**：route 整合測試 import app 時會連帶執行
  `db.ts`（建目錄、開檔）。→ 用 `DB_PATH` 指向 `os.tmpdir()` 下暫存檔，測試後刪除；
  確保不在 repo 內留下 `data/`。
- **route 直接 import db 單例，難以乾淨注入**：→ 本 issue 用 `DB_PATH` 隔離先讓測試成立；
  #03 完成 db 注入後再回來簡化（已在「開放問題」標記，建議寫進 #03 plan）。
- **mock 漂移**：mock 的外部介面與真實 `lib/github` 等簽章不一致時測試會假綠。
  → mock 緊貼真實函數簽章，並在型別層用 `typeof` 對齊。
- **Bun test 與 D1 無關**：本 issue 不測 D1；#02 換 driver 後需另補 D1 路徑測試（標記給 #02）。

### Notes for the implementer

- **先讀後改**：動 `db.ts` 抽 `applySchema` 前先 grep 所有 import `db.ts` 的地方，
  確認抽離後 side effect 順序不變（建目錄 → 建表 → migration）。
- 測試目錄慣例擇一即定，不要 `__tests__` 與 `test/` 混用。
- 外部依賴 mock 用 `bun:test` 的 `mock.module()`；放在測試檔頂端、import 受測模組之前。
- repo 測試的建表務必包含 #01 migration 加的欄位（`github_path` / `github_sha` / `slug`），
  否則 slug 系列測試會炸。
- 依專案規範：若為可測性微調了函式行為（如抽 `applySchema`），同步更新該函式 JSDoc，
  `@remarks` 記錄「為何抽離（測試與 production 共用 DDL 單一來源）」。
- CI workflow 用官方 `oven-sh/setup-bun@v2`，`working-directory: backend`，
  cache `~/.bun/install/cache` 加速。
